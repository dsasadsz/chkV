# Architecture

## Суть проекта

`Travel X` - учебный Django-проект сайта космического туризма. Сейчас это не статический сайт и не CGI-приложение, а полноценный Django-проект с регистрацией, входом, личным кабинетом, загрузкой фотографии профиля, публичными страницами и закрытой страницей администратора.

Главная идея проекта:

- обычный пользователь видит публичный сайт `Travel X`, может смотреть направления, регистрироваться, входить в аккаунт и загружать фото профиля;
- авторизованный пользователь видит свое имя и аватар в правой части навигации;
- администратор дополнительно получает доступ к конфиденциальной странице `/staff-home/` и стандартной Django admin-панели `/admin/`;
- данные пользователей хранятся в SQLite, а фотографии профиля сохраняются в `media/profile_photos/user_<id>/...`.

## Стек

- Python 3.11
- Django 5
- SQLite
- Django Templates
- Django Auth
- Pillow для проверки загружаемых изображений
- HTML, CSS, JavaScript
- PowerShell / PyCharm для локального запуска

Основные зависимости описаны в `requirements.txt`.

## Основные маршруты

- `/` - публичная главная страница.
- `/destinations/` - публичная страница направлений с карточками туров, тестом и бронированием.
- `/register/` - регистрация пользователя.
- `/success/` - страница успешной регистрации.
- `/login/` - вход в аккаунт.
- `/dashboard/` - личный кабинет пользователя.
- `/dashboard/photo/` - POST-обработчик обновления фото профиля.

## Django project: `travelx_project/`

`travelx_project/` - корневой Django-проект.

Важные файлы:

- `settings.py` - настройки Django: приложения, шаблоны, база данных, статика, медиа, язык, часовой пояс.
- `urls.py` - корневые URL. Подключает `/admin/` и маршруты приложения `accounts`.
- `wsgi.py` и `asgi.py` - стандартные точки входа Django.

В `settings.py` подключены:

- `accounts`;
- папка шаблонов `templates`;
- папка статики `static`;
- папка медиа `media`;
- контекст-процессор `accounts.context_processors.nav_profile`.

Контекст-процессор нужен, чтобы в любом шаблоне можно было показать имя и аватар текущего пользователя.

## Django app: `accounts/`

`accounts/` - главное приложение проекта. Несмотря на название, оно отвечает не только за аккаунты, но и за публичные страницы сайта.

Важные файлы:

- `models.py` - модель профиля пользователя.
- `forms.py` - формы регистрации, входа и загрузки фото.
- `views.py` - вся логика страниц.
- `urls.py` - маршруты приложения.
- `admin.py` - регистрация модели профиля в Django admin.
- `signals.py` - автоматическое создание профиля при создании пользователя.
- `context_processors.py` - данные пользователя и аватара для навигации.
- `tests.py` - тесты основных сценариев.
- `migrations/0001_initial.py` - миграция модели профиля.

### Модель данных

Используется стандартная модель Django:

- `django.contrib.auth.models.User`

Дополнительная модель:

- `UserProfile`

`UserProfile` связан с `User` через `OneToOneField`.

Поля `UserProfile`:

- `user` - связь с пользователем;
- `age` - возраст пользователя;
- `profile_image` - фото профиля;
- `created_at` - дата создания профиля;
- `updated_at` - дата обновления профиля.

Фото сохраняется функцией `profile_image_upload_to()` по схеме:

```text
media/profile_photos/user_<id>/<random_uuid>.<extension>
```

Это важно: фотографии не складываются в одну общую папку, а разделяются по пользователям.

### Формы

`forms.py` содержит:

- `SignUpForm` - регистрация пользователя, email, возраст, необязательное фото профиля;
- `StyledAuthenticationForm` - форма входа;
- `ProfileImageForm` - обновление фото профиля;
- `validate_uploaded_image()` - проверка, что загруженный файл действительно является изображением.

Поддерживаемые типы изображений:

- `image/jpeg`
- `image/png`
- `image/webp`
- `image/gif`

### Представления

`views.py` содержит:

- `home()` - публичная главная;
- `destinations_view()` - направления и тест;
- `registration_success_view()` - успешная регистрация;
- `staff_home()` - закрытая страница администратора;
- `register_view()` - регистрация;
- `login_view()` - вход;
- `dashboard_view()` - личный кабинет;
- `update_profile_photo()` - обновление фото;
- `logout_view()` - выход.

В `views.py` также лежат данные для публичной части:

- `TRAVEL_FEATURES`
- `TRAVEL_DESTINATIONS`
- `QUIZ_QUESTIONS`

Это простые Python-списки. Если проект будет расти, их можно вынести в модели базы данных.

## Пользовательские сценарии

### Регистрация

1. Пользователь открывает `/register/`.
2. Заполняет имя, email, возраст, пароль.
3. Может загрузить фото профиля.
4. Django создает `User`.
5. Через сигнал создается `UserProfile`.
6. Форма записывает возраст и фото в профиль.
7. Пользователь автоматически входит в аккаунт.
8. Его отправляет на `/success/`.

### Вход

1. Пользователь открывает `/login/`.
2. Вводит логин и пароль.
3. Django авторизует пользователя.
4. После входа пользователь попадает в `/dashboard/` или на URL из `next`.

### Личный кабинет

1. Пользователь открывает `/dashboard/`.
2. Видит имя, email, возраст, дату регистрации, фото профиля.
3. Может загрузить новое фото.
4. Старое фото удаляется, новое сохраняется в папку пользователя.

### Закрытая страница администратора

1. Администратор открывает `/staff-home/`.
2. Если `user.is_staff=True`, страница открывается.
3. Если обычный пользователь, Django возвращает `403 Forbidden`.
4. Гость будет перенаправлен на страницу входа.

## Что важно знать ассистенту

- Главная рабочая логика находится в `accounts/`.
- Публичная часть сайта не статическая: она собрана через Django templates.
- Файлы в `media/` - пользовательские загрузки.
- `db.sqlite3` - локальная база разработки, ее не надо удалять без явной просьбы.
- `__pycache__/` можно игнорировать.
- `.idea/` относится к PyCharm и не влияет на работу Django.
- Если появляются старые HTML/CGI-файлы, они относятся к старой версии и не должны быть главным источником правок.
- При изменении маршрутов сначала смотреть `accounts/urls.py`, потом `accounts/views.py`.
- При изменении данных профиля сначала смотреть `accounts/models.py`, `accounts/forms.py`, `accounts/signals.py`.
- При изменении шапки и аватара смотреть `accounts/context_processors.py`, `templates/includes/nav_user.html`, `templates/base.html`, `templates/public_base.html`.
- При изменении дизайна публичных страниц смотреть `static/css/public_site.css` и `static/js/public_site.js`.
- При изменении кабинета и внутренних страниц смотреть `static/css/styles.css` и `static/js/app.js`.

## Проверка после изменений

Минимальная проверка:

```powershell
py -3.11 manage.py check
py -3.11 manage.py test
```

Ручная проверка в браузере:

- открыть `/`;
- открыть `/destinations/`;
- зарегистрировать нового пользователя;
- проверить `/success/`;
- проверить `/dashboard/`;
- загрузить фото профиля;
- войти под admin и открыть `/staff-home/`;
- открыть `/admin/`.

## Известные особенности

- Проект учебный, `DEBUG=True`.
- `SECRET_KEY` находится прямо в `settings.py`, для production так делать нельзя.
- `ALLOWED_HOSTS` настроен для локальной разработки.
- Часть текстов в старых файлах могла иметь проблемы с кодировкой. Новые документы и шаблоны лучше сохранять в UTF-8.
- Данные направлений и теста сейчас лежат в `accounts/views.py`, а не в базе данных.
